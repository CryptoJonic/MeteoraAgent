Работай автономно, но не смешивай исследование стратегии с текущей рабочей paper-логикой без явного экспериментального режима.

## Критически важное определение события

Для каждой галки есть уровень `L`.

Нормализованная глубина под уровнем:

```text
depth_pct = (L - low_after_activation) / L * 100
```

Успешный возврат:

```text
цена после ухода ниже L снова торгуется на L или выше
```

Для режима reclaim:

```text
цена достигает L * (1 + reclaim_buffer_pct / 100)
```

Нужно различать:

1. глубину до первого возврата;
2. максимальную adverse excursion до возврата;
3. максимальную favorable excursion после возврата;
4. время до первой лимитки;
5. время до максимальной глубины;
6. время до возврата;
7. время под уровнем;
8. возврат в заданном горизонте;
9. отсутствие возврата;
10. данные, оборванные концом выборки — censored, а не автоматически loss.

Не анализируй только успешные галки. Это создаёт survivorship bias. Для каждой глубины нужно показывать:

```text
P(return within H | depth D was reached)
```

Именно эта условная вероятность нужна для решения, продолжать ли усреднение или фиксировать убыток.

## Поиск кандидатов галок

Не ограничивайся одной жёсткой формулой текущего Radar.

Сначала создай широкий candidate detector с высокой полнотой. Он может давать лишние кандидаты, которые затем классифицируются и фильтруются.

Признаки кандидата должны включать минимум:

### Геометрия

- глубина левого движения в процентах и ATR;
- глубина правого восстановления;
- длина левого плеча;
- длина правого плеча;
- баланс плеч;
- угол/скорость падения;
- угол/скорость восстановления;
- острота минимума;
- ширина основания;
- число свечей возле минимума;
- одна свеча или серия минимумов;
- wick/body характеристики;
- положение close относительно low;
- наличие второго теста;
- число предыдущих касаний уровня;
- возраст уровня;
- был ли уровень уже частично «съеден».

### Контекст рынка

- ATR и percentile волатильности;
- объём и volume spike;
- slope локального тренда;
- положение относительно EMA/VWAP;
- расстояние от локального high;
- предыдущий импульс;
- боковой, восходящий или нисходящий режим;
- время суток;
- день недели;
- одновременное движение BTC, ETH и SOL;
- синхронность галок между инструментами;
- крупный общий market sell-off против локальной галки.

Не добавляй сотни бессмысленных индикаторов. Каждый признак должен быть интерпретируемым и иметь проверяемую гипотезу.

## Типы галок

Разбей галки на небольшое число устойчивых и понятных типов.

Целевой диапазон:

```text
4–7 типов
```

Не придумывай типы заранее только по визуальному впечатлению. Используй двухэтапный подход:

1. статистическое clustering / representation;
2. перевод кластеров в понятные человеку названия и правила.

Допускаются, если данные подтверждают, примеры названий:

- Fast V / быстрый возврат;
- Deep capitulation / глубокий выброс;
- Rounded recovery / округлое восстановление;
- Asymmetric weak / слабая асимметричная;
- Multi-test / повторно тестируемая;
- Consumed level / уже съеденный уровень;
- Trend trap / галка против сильного тренда.

Но итоговые типы должны появиться из данных, а не из этого списка.

Для каждого типа подготовь:

- формальное описание;
- типичный пример;
- медианный профиль цены;
- распределение глубины;
- вероятность возврата;
- время возврата;
- вероятность провала;
- MFE после возврата;
- лучшие и худшие рыночные режимы;
- частоту появления;
- статистику отдельно по BTC, ETH, SOL;
- статистику отдельно по таймфреймам;
- long-term и recent статистику;
- sample size;
- confidence intervals.

## Недавнее поведение

Пользователю особенно важно понимать, куда цена падает последнее время.

Добавь rolling статистику:

- 7 дней;
- 30 дней;
- 90 дней;
- 180 дней;
- 365 дней;
- вся история.

И recency-weighted статистику, где новые события имеют больший вес.

На графике должно быть видно:

```text
Исторически p75 = 0.92%
За 90 дней p75 = 1.34%
В текущем volatility regime p75 = 1.48%
```

Это важнее, чем одно вечное среднее по всей истории.

## Сетка лимиток

Не оптимизируй сетку только по максимальной прибыли на истории.

Нужно построить несколько прозрачных профилей:

### Conservative

- меньший effective leverage;
- основной капитал в неглубокой зоне;
- резерв ниже;
- ограничение максимального убытка в деньгах.

### Balanced

- распределение по статистической плотности глубин;
- риск и ожидаемая прибыль сбалансированы.

### Aggressive paper-only

- более глубокая загрузка;
- только для paper и stress test;
- не использовать как рекомендацию для реальных денег.

Для каждого типа галки и режима рынка рассчитай:

- рекомендуемый диапазон сетки;
- число лимиток;
- шаг;
- веса;
- средний вход при каждом количестве fills;
- PnL на возврате к GALKA;
- PnL на reclaim;
- вероятность полного заполнения;
- риск при продолжении падения;
- ожидаемый drawdown;
- CVaR / tail risk;
- приблизительный liquidation buffer при заданном effective leverage;
- максимальный риск в долларах.

Термин `ликвиднее размещать лимитки` в интерфейсе замени на более точный:

```text
статистически плотная зона входов
```

Это распределение исторических глубин, а не реальная биржевая ликвидность order book.

## Stop / invalidation

Главный вопрос: когда перестать ждать возврат.

Не выбирай stop по одному максимальному историческому провалу.

Построй условные кривые:

```text
P(return within H | current depth D, galka type T, regime R)
```

Покажи точки, где:

- вероятность возврата резко падает;
- ожидаемое время возврата резко растёт;
- expected value после комиссий становится отрицательным;
- tail risk становится непропорциональным прибыли;
- stop в деньгах достигает заданного лимита.

Сравни минимум:

- fixed depth stop;
- percentile stop;
- ATR stop;
- time stop;
- probability stop;
- hybrid stop: depth + time + regime;
- no-stop paper stress test только как опасный benchmark.

Вывод должен быть не `идеальный стоп`, а таблица trade-off.

## Выход и удержание роста

Сравни:

- закрытие ровно на GALKA;
- reclaim +0.10% и trail 0.75%;
- разные reclaim buffer;
- trail 0.15 / 0.30 / 0.50 / 0.75 / 1.0%;
- ATR trailing;
- trailing под локальными минимумами;
- частичное закрытие на GALKA и runner;
- ограничение maximum hold time.

Не меняй production paper defaults автоматически. Все новые варианты должны быть отдельными исследовательскими профилями.

## Anti-overfit

Обязательные правила:

- никаких случайных train/test split для временных рядов;
- chronological split;
- walk-forward validation;
- отдельный final OOS, который не используется для настройки;
- параметры выбираются на train/validation и один раз проверяются на OOS;
- не оптимизировать отдельно под одну красивую сделку;
- не смешивать результат BTC, ETH, SOL так, чтобы один инструмент скрывал провал другого;
- bootstrap confidence intervals;
- minimum sample threshold;
- multiple-testing warning;
- report всех неудачных вариантов, а не только победителя;
- model/version hash для каждого отчёта;
- воспроизводимый seed;
- явная проверка data leakage.

## Shadow mode

Добавь shadow режим:

- Radar находит галку;
- бот не торгует;
- система записывает, что произошло бы при выбранном профиле;
- сохраняет fills, MAE, reclaim, trailing и итог;
- сравнивает radar decision с ручным выбором пользователя;
- строит live forward статистику;
- не влияет на настоящий paper balance.

Это обязательный мост между историческим backtest и auto-paper.

## Этапы автоматизации

### Phase A — Manual Galka

Пользователь выбирает GALKA. Алгоритм исполняет сетку и выход.

### Phase B — Assisted Radar

Radar предлагает тип, score, depth bands и profile. Пользователь нажимает `Торговать` или `Пропустить`.

### Phase C — Shadow Auto

Radar автоматически создаёт виртуальные shadow trades, но не влияет на paper balance.

### Phase D — Restricted Auto-Paper

Только типы и режимы, которые прошли OOS и live shadow thresholds, могут автоматически открывать отдельные auto-paper кампании.

Обязательные ограничения:

- allowlist типов;
- minimum confidence;
- minimum sample size;
- maximum simultaneous risk;
- per-symbol cap;
- daily loss cap;
- kill switch;
- no averaging below last configured limit;
- no hidden parameter changes;
- version shown on every trade.

### Phase E — Real trading

Не реализовывать в этом задании.

Реальные ордера потребуют отдельной архитектуры, security review, explicit user confirmation, risk controls и отдельного PR.

## Success criteria

Работа считается завершённой, когда:

1. вся доступная история воспроизводимо обработана;
2. все кандидаты галок имеют features и outcomes;
3. получено 4–7 интерпретируемых типов либо честно доказано, что устойчивых типов меньше;
4. для каждого типа есть depth, return, time и risk distributions;
5. recent статистика отделена от full-history;
6. на графике можно быстро открыть статистику любой галки;
7. приложение показывает sample size и uncertainty;
8. shadow mode собирает forward результаты;
9. auto-paper остаётся выключенным по умолчанию;
10. `galka-pro-v1` и действующие paper-позиции сохраняются;
11. `npm run check` проходит;
12. отдельные research tests проходят;
13. PR содержит отчёты, screenshots и инструкции воспроизведения;
14. никакие реальные ордера не добавлены.
