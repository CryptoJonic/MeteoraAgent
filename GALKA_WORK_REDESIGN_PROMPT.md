## Текущее назначение приложения

Galka Pro — мобильный терминал для:

1. просмотра BTCUSDT, ETHUSDT и SOLUSDT по публичным данным Binance USD-M Futures;
2. ручного выбора уровня «галки»;
3. автоматического построения лестницы paper-лимиток ниже уровня;
4. paper-исполнения лимиток;
5. возврата к уровню и выхода через reclaim + trailing-stop;
6. сохранения выбранных пользователем галок как обучающих примеров;
7. визуального радара кандидатов галок;
8. рисования и анализа графика.

Это paper-продукт. Он не должен использовать API-ключи и не должен отправлять реальные биржевые ордера.

## Неприкосновенные требования

### 1. Не потерять paper-состояние

Ключ браузерного хранилища:

```text
galka-pro-v1
```

Его нельзя менять без полноценной миграции. После обычного `git pull` и перезапуска Termux должны сохраняться:

- уровни GALKA по BTC, ETH и SOL;
- все ожидающие лимитки;
- заполненные уровни;
- открытые paper-позиции;
- trailing-stop и его состояние;
- история сделок;
- настройки;
- рисунки;
- ручные обучающие примеры.

Любое изменение структуры store выполняй через совместимую миграцию и добавь автоматическую проверку миграции.

### 2. Не менять торговую методику скрытно

Существующая логика long-only должна остаться неизменной, пока изменение не выделено отдельным экспериментом:

- пользователь вручную ставит уровень GALKA;
- лимитки строятся ниже уровня с заданным шагом и глубиной;
- после первой покупки уровень блокируется;
- при возврате выше reclaim-уровня включается trailing-stop;
- trailing-stop не опускается ниже уровня GALKA;
- радар только подсвечивает кандидатов и не открывает сделки;
- paper-бот обслуживает три инструмента одновременно.

Не смешивай редизайн с оптимизацией стратегии.

### 3. Сохранить простой запуск в Termux

Команда пользователя должна оставаться простой:

```bash
cd ~/Galka
git pull --ff-only origin main
bash scripts/start-termux.sh
```

Терминал должен открываться на Android без обязательного `npm install` и без ручной сборки.

Можно модернизировать архитектуру, разбить JavaScript на модули и использовать build tooling только при условии, что:

- готовый production build хранится в репозитории или собирается автоматически;
- Termux по-прежнему запускает всё одной командой;
- приложение не зависит от мощного компьютера;
- нет секретов и серверной инфраструктуры.

## Основная UX-цель

График — главный объект. В обычном режиме ничто не должно перекрывать свечи без явного действия пользователя.

Интерфейс должен ощущаться как современный премиальный мобильный терминал, но не как дешёвая копия TradingView.

Направление:

- тёмная нейтральная основа;
- высокая контрастность свечей и торговых уровней;
- минимум декоративного неона;
- цвет используется функционально;
- аккуратная типографика;
- крупные touch-targets;
- минимум постоянных панелей;
- плавные, короткие анимации;
- визуальная иерархия вместо множества рамок;
- отсутствие фоновой сетки по умолчанию;
- никаких лишних строк OHLC, статуса или диапазонов, если они не вызваны пользователем.

## Новая структура интерфейса

### Верхняя компактная панель

Оставить только важное:

- инструмент BTC / ETH / SOL;
- таймфрейм;
- состояние соединения в виде маленькой точки/иконки;
- кнопка Radar;
- меню дополнительных действий.

Не занимать две-три строки.

### Полноэкранный график

На графике должны быть максимально понятны:

- текущая цена;
- GALKA;
- L1…Ln;
- заполненные лимитки;
- средняя цена;
- reclaim;
- trailing-stop;
- входы и выходы;
- radar-кандидаты;
- выбранный radar-кандидат.

Нужен умный контроль наложений подписей: ценники не должны закрывать друг друга. На узком экране допускается группировка/сворачивание близких меток.

### Нижняя навигация

Сделать 4–5 понятных разделов:

- Chart;
- Paper;
- Radar;
- Watchlist;
- More.

Панели открываются снизу как удобные bottom sheets с несколькими фиксированными высотами. Они должны поддерживать свайп, закрытие вниз и не ломать масштаб графика.

### Быстрые действия на графике

Добавить плавающую компактную кнопку или контекстное меню:

- поставить GALKA;
- ввести точную цену;
- двигать GALKA;
- включить/выключить Radar;
- скрыть/показать уровни;
- Fit;
- перейти к текущей цене.

Не дублировать эти действия в трёх разных местах.

## Переработка Paper-панели

Панель должна сразу отвечать на вопросы:

- по какой монете сейчас стоят лимитки;
- есть ли заполнения;
- какая средняя цена;
- сколько номинала использовано;
- текущий PnL;
- где reclaim;
- активирован ли trailing;
- где stop;
- когда истекает кампания;
- работает ли поток котировок.

Сделать отдельные карточки по BTC, ETH и SOL, чтобы состояние всех трёх инструментов было видно без постоянного переключения графика.

Для лестницы:

- компактный список уровней;
- WAIT / FILLED / CANCELLED;
- цена, глубина, номинал;
- визуальная полоса заполнения;
- общий счётчик;
- возможность свернуть список.

Добавить безопасный pre-trade preview до подтверждения нового ручного уровня:

- уровень GALKA;
- первая и последняя лимитка;
- количество ступеней;
- общий номинал;
- примерная средняя цена при полном заполнении;
- ориентировочный PnL при возврате к GALKA;
- предупреждение, что это paper.

## Переработка Radar

Radar остаётся визуальным помощником и не торгует автоматически.

Нужно:

- подсвечивать кандидаты только на текущем инструменте и таймфрейме;
- фильтр: все / сильные / средние;
- настройка минимального score;
- режим «только видимый участок»;
- счётчик найденных галок;
- тап по кандидату открывает карточку объяснения;
- карточка показывает score, drop ATR, recovery, баланс плеч, sharpness и close lift;
- кнопки `Это галка` и `Не галка`;
- сохранять обе оценки как обучающие данные;
- отдельно отмечать совпадение с ручным выбором пользователя;
- быстро перейти к предыдущей/следующей найденной галке;
- не засорять график десятками меток при сильном уменьшении масштаба — использовать кластеризацию.

Важно: это пока explainable scoring model, не называй её обученной нейросетью.

## Новые полезные функции

Реализуй только функции, которые дают практическую пользу и не перегружают экран.

Приоритетные:

### 1. Session Health

Компактный индикатор:

- WebSocket online/offline;
- возраст последней котировки;
- активен ли browser tab;
- сколько времени paper-движок не получал данные;
- предупреждение после восстановления соединения.

После реконнекта приложение должно запросить пропущенные свечи через REST, сверить состояние и продолжить работу без двойных исполнений.

Не обещай полноценную фоновую торговлю Android, если браузер заморожен. Вместо этого честно показывай состояние сессии.

### 2. Backup / Restore

Одна кнопка экспортирует полный локальный snapshot:

- paper;
- settings;
- drawings;
- training examples;
- radar labels.

Импорт должен сначала показывать, что будет восстановлено, и создавать резервную копию текущего состояния.

### 3. Installable PWA

Добавить manifest, иконки и service worker для установки на домашний экран и кэширования статических файлов.

Service worker не должен имитировать работу paper-бота в фоне. Он нужен для надёжного запуска интерфейса и загрузки ресурсов.

### 4. Onboarding

Короткий интерактивный запуск из 4–5 шагов:

- выбрать монету;
- поставить GALKA;
- проверить лестницу;
- понять reclaim/trailing;
- включить Radar.

Он показывается один раз и доступен повторно из справки.

### 5. Activity log

Хронологический журнал:

- уровень установлен/перенесён;
- лимитка заполнена;
- trailing активирован;
- stop поднят;
- сделка закрыта;
- соединение потеряно/восстановлено;
- импорт/экспорт.

### 6. Replay для обучения

Улучшить существующий Replay:

- скрывать будущее;
- шаг по одной свече;
- пользователь отмечает галку;
- после прохождения участка показывать, как цена отработала;
- сохранять результат как размеченный пример;
- не использовать будущие свечи при первоначальной оценке кандидата.

Paper engine должен быть максимально детерминированным и отделённым от DOM, чтобы его можно было тестировать на последовательности котировок.

## Надёжность и безопасность

Обязательные инварианты:

- одна и та же котировка не заполняет уровень дважды;
- уровни заполняются только при достижении цены;
- после первого заполнения GALKA нельзя двигать;
- trailing-stop никогда не снижается;
- trailing-stop не ниже GALKA;
- закрытая кампания не обрабатывается повторно;
- реконнект не создаёт дубликат сделки;
- radar не вызывает `createCampaign`;
- изменение UI не меняет paper state;
- reset требует явного подтверждения;
- импорт не уничтожает текущие данные без backup;
- никакие API keys/secret keys не должны появляться в клиенте.

Работа считается принятой, когда:

- интерфейс визуально заметно лучше текущего;
- график занимает максимум полезной площади;

## Принцип принятия решений

Не копируй TradingView целиком. Бери только полезные паттерны взаимодействия. Galka должна иметь собственную идентичность: простой терминал для одной конкретной методики, где уровень, лестница, возврат, trailing и radar понятнее, чем в универсальной биржевой платформе.

Сначала сохраняй надёжность и данные, затем улучшай UX, затем добавляй функции. Красота без безопасности paper-состояния не считается результатом.
